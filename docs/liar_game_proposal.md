# 1. 프로젝트 개요

## 1.1 게임 콘셉트 및 비전

**프로젝트명**: 라이어 게임 온라인 (가칭)

**핵심 콘셉트**  
점심시간 10분이면 충분한, 웹 URL만으로 친구들과 즉시 즐기는 소셜 추리 파티 게임. 다운로드 없이 브라우저에서 바로 플레이하며, **진실과 거짓 사이의 긴장감**을 캐주얼하게 즐긴다.

**비전**  
Among Us 이후 침체된 소셜 디덕션 장르에 **접근성의 혁명**을 일으킨다. 앱 다운로드라는 장벽을 없애고, 누구나 링크 하나로 모임을 시작할 수 있는 "파티 게임의 카카오톡"이 된다.

**게임 장르**: 소셜 디덕션 / 추리 / 파티 게임  
**플랫폼**: 웹 브라우저 (PC/모바일 반응형)  
**플레이 인원**: 4-10명 (최적 6-8명)  
**플레이 타임**: 1라운드 8-12분

## 1.2 핵심 가치 제안

**기존 게임들의 한계**
- 마피아42: 앱 다운로드 필수, 초보자 진입장벽 높음
- Among Us: 다운로드 필요, 음성 채팅 미지원
- 웹마피아: 구식 UI, 낮은 모바일 최적화

**우리의 차별점**
1. **제로 프릭션 진입**: URL만으로 3초 만에 게임 시작
2. **하이브리드 커뮤니케이션**: 텍스트 + 선택적 음성
3. **점진적 학습 곡선**: 2분 튜토리얼 후 바로 재미
4. **크로스 플랫폼**: PC와 모바일 동시 플레이
5. **건전한 수익화**: P2W 없는 코스메틱 중심

# 4. 게임 시스템 설계

## 4.1 핵심 게임 룰

**역할**
- **시민**: 공통 제시어 받음, 라이어 찾기
- **라이어**: 제시어 모름, 정체 숨기기

**승리 조건**
- 시민: 투표로 라이어 찾기
- 라이어: 생존 또는 제시어 맞추기

## 4.2 게임 플로우

```
역할 배정 (10초) → 토론 (3분) → 투표 (1분) → 결과 (30초)
→ 3라운드 반복 → 최종 결과
```

## 4.3 득점 시스템

| 상황 | 시민 점수 | 라이어 점수 |
|------|----------|------------|
| 라이어 찾기 성공 + 제시어 실패 | +1점 | 0점 |
| 라이어 찾기 성공 + 제시어 성공 | 0점 | +3점 |
| 라이어 찾기 실패 | 0점 | +2점 |

# 8. 기술 아키텍처

## 8.1 전체 스택

```
Frontend: Next.js 15 (SSR/CSR Hybrid) + Socket.IO Client
Backend: NestJS 11 + Fastify (WebSocket)
Database: Redis (Cache/Session) + PostgreSQL (Persistent)
Infrastructure: Kubernetes on Mac mini M4
```

## 8.2 Next.js 활용 전략

**SSR/CSR 하이브리드**
- 홈/랭킹 페이지: SSR (SEO)
- 게임 화면: CSR (`"use client"`)
- WebSocket 연결: 클라이언트 사이드만

**성능 최적화**
- Turbopack 빌드 (15.5+)
- Server Components (무거운 로직)
- Image 최적화 (WebP/AVIF)
- Code Splitting

**배포**
- Custom Node.js 서버 필수 (WebSocket)
- Vercel 불가 → AWS Fargate/Railway

## 8.3 NestJS + Fastify 백엔드

**아키텍처**
```typescript
@Module({
  imports: [
    GameModule,      // 게임 로직
    MatchModule,     // 매칭 시스템
    UserModule,      // 사용자 관리
    ChatModule,      // 채팅/음성
  ]
})
```

**WebSocket Gateway**
```typescript
@WebSocketGateway({ cors: true })
export class GameGateway {
  @SubscribeMessage('playerAction')
  handleAction(client: Socket, payload: any) {
    // 게임 상태 업데이트
    this.gameService.processAction(payload);
    // 모든 플레이어에게 브로드캐스트
    this.server.to(roomId).emit('gameUpdate', state);
  }
}
```

**Fastify 성능**
- Express 대비 2-3배 빠름
- 요청당 ~2ms (vs Express ~30ms)

## 8.4 실시간 통신

**Socket.IO**
- 양방향 통신
- 자동 재연결
- Room 기반 브로드캐스트

**WebRTC (음성)**
- P2P 연결 (서버 부하 감소)
- STUN/TURN 서버 (coturn)
- Fallback: 서버 중계

## 8.5 데이터베이스 설계

**Redis (Hot Data)**
```
게임 세션: game:session:{roomId}
플레이어 상태: player:{userId}:status
리더보드: leaderboard:global (Sorted Set)
매칭 큐: matchmaking:queue (List)
```

**PostgreSQL (Cold Data)**
```sql
-- 사용자 테이블
CREATE TABLE users (
  id UUID PRIMARY KEY,
  username VARCHAR(50),
  email VARCHAR(255),
  level INT DEFAULT 1,
  created_at TIMESTAMP
);

-- 게임 히스토리
CREATE TABLE game_history (
  id UUID PRIMARY KEY,
  room_id VARCHAR(10),
  players JSONB,
  result JSONB,
  created_at TIMESTAMP
);

-- 랭킹
CREATE TABLE rankings (
  user_id UUID,
  season INT,
  rating_points INT,
  tier VARCHAR(20),
  PRIMARY KEY (user_id, season)
);
```

**데이터 흐름**
```
쓰기: App → Redis (즉시) → PostgreSQL (비동기)
읽기: App → Redis (캐시) → PostgreSQL (미스 시)
```
